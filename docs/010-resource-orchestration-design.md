# NOVA 资源编排设计

## 一、设计背景

根据当前平台规划，NOVA 不仅负责 CI/CD 流程治理，还需要承担一部分“服务上线所需资源自动生成”的能力。

已知前提包括：

- 当前只有一套内网 Kubernetes 集群
- 运行基础为腾讯云 CVM
- CI 跑在集群中的 Gitea Runner
- CD 上线前往往需要先申请域名、PVC 等资源
- 腾讯云 Terraform 已经打通
- 集群内的 HTTPRoute 和 PVC 自动创建还未建设
- 期望开发者只填写域名和卷等需求，平台自动生成资源

因此，NOVA 需要提供一个面向“服务上线资源”的自动编排能力。

## 二、设计目标

资源编排模块的目标是：

- 将资源申请纳入标准交付流程
- 让开发者只声明资源需求，不手写 K8s 清单
- 由平台自动生成并管理基础资源
- 将资源变更纳入治理和审批
- 与 Terraform、Kubernetes 保持清晰边界
- 为后续扩展更多资源类型保留空间

## 三、编排边界

### NOVA 负责

- 接收服务资源声明
- 生成 Kubernetes 内部资源
- 管理资源申请状态
- 记录资源变更历史
- 协调是否需要审批
- 触发 Terraform 或平台内部渲染逻辑

### Terraform 负责

- 云侧资源编排
- 腾讯云相关 DNS、基础设施能力
- 云资源变更与创建

### Kubernetes 负责

- Deployment
- Service
- HTTPRoute
- PVC
- Namespace
- ConfigMap
- Secret（后续可扩展）

## 四、资源分类

建议将资源分为两类：

### 1. 平台内资源

直接由 NOVA 在 K8s 中生成或 apply 的资源，例如：

- Namespace
- Service
- HTTPRoute
- PVC
- Deployment 配套资源
- ConfigMap

### 2. 云侧资源

通过 Terraform 管理的资源，例如：

- 域名解析
- 云负载入口相关配置
- 未来可能的存储、证书等云资源

## 五、开发者输入模型

开发者不直接提交 K8s YAML，而是在 WebUI 或 `nova.yml` 中声明资源需求。

### 典型配置

```yaml
resources:
  network:
    domain: iplocator.internal.example.com
    port: 3000
  storage:
    - name: uploads
      size: 10Gi
      mountPath: /app/uploads
```

开发者只需要表达：

- 服务端口
- 访问域名
- 卷名称
- 卷大小
- 挂载路径

平台负责将其翻译为可执行资源。

## 六、资源生成对象

一期建议支持自动生成以下对象：

### 1. Namespace

命名规则：

```text
{env}-{project}-{service}
```

例如：

```text
dev-app-iplocator
staging-app-iplocator
prod-app-iplocator
```

### 2. Service

根据服务端口自动生成 ClusterIP Service。

### 3. HTTPRoute

若填写域名，则自动生成 HTTPRoute 规则，将域名流量导向对应 Service。

### 4. PVC

若声明卷，则按卷配置生成 PVC。

### 5. ConfigMap（可选）

后续可将部分平台渲染后的环境配置写入 ConfigMap。

## 七、资源生成时机

建议区分以下时机：

### 1. 新服务首次接入时

- 创建基础配置
- 生成 dev / staging / prod namespace（可按需创建）
- 若填写域名和卷，则生成对应资源申请记录

### 2. 首次部署前

- 若目标环境所需资源不存在，则自动创建
- 若资源需要审批，则等待审批后再创建

### 3. 配置变更后

- 若 `nova.yml` 中资源配置发生变化，平台识别为“资源变更”
- 可生成资源变更记录
- 按策略决定是否需要审批

## 八、资源变更策略

需要区分两类情况：

### 1. 仅镜像更新

例如：

- 更新 Deployment 的镜像 Tag
- 不涉及域名、PVC、端口等资源变化

这种情况不应走资源审批。

### 2. 资源配置变化

例如：

- 新增域名
- 新增卷
- 扩容 PVC
- 修改服务端口
- 新增路由

这种情况建议视为资源变更，进入治理流程。

## 九、资源变更审批建议

一期可以采用简化策略：

- 新服务首次申请资源：需要审批
- 已有服务仅镜像更新：不需要资源审批
- 若资源声明发生变化：触发资源变更审批

这样既符合治理要求，又不会让普通迭代发布变得过重。

## 十、资源编排流程

### 1. 新服务上线流程

```text
开发者填写服务配置
        ↓
NOVA 校验 `nova.yml`
        ↓
识别资源声明
        ↓
生成资源申请单
        ↓
审批通过
        ↓
创建 Namespace / Service / HTTPRoute / PVC
        ↓
允许部署继续执行
```

### 2. 已有服务迭代上线流程

```text
镜像版本更新
        ↓
资源无变化
        ↓
直接进入部署流程
```

### 3. 已有服务资源变更流程

```text
修改域名或卷配置
        ↓
NOVA 识别为资源变更
        ↓
创建资源变更审批
        ↓
审批通过
        ↓
执行资源变更
```

## 十一、Kubernetes 资源模板建议

建议在中央模板仓库中维护以下模板：

- `deployment.yaml.j2`
- `service.yaml.j2`
- `httproute.yaml.j2`
- `pvc.yaml.j2`

平台根据服务配置渲染这些模板。

### 设计原则

- 开发者不维护这些模板
- 模板统一由平台治理
- 所有资源命名规则统一
- 模板中可插入环境、服务名、端口、域名、卷等变量

## 十二、Terraform 联动建议

由于腾讯云 Terraform 已打通，平台后续可在资源编排模块中增加“云侧资源适配器”。

### 一期建议

一期不强行把所有资源都放进 Terraform，而是区分：

- K8s 内部资源：平台直接创建
- 云侧资源：平台触发 Terraform 流程

### 可联动场景

- 域名解析记录创建
- 特定云资源申请
- 未来公网暴露相关资源

## 十三、资源状态管理

建议平台为资源申请和资源变更维护状态。

### 状态建议

- `pending`
- `approved`
- `rejected`
- `applying`
- `applied`
- `failed`

### 资源记录内容建议

- 服务 ID
- 环境
- 资源类型
- 资源名称
- 配置快照
- 当前状态
- 审批信息
- 执行时间
- 错误信息

## 十四、失败处理策略

资源编排失败时需要区分阶段：

### 1. 审批前失败

通常为配置校验失败，应直接提示用户修正。

### 2. apply 阶段失败

例如：

- PVC 参数不合法
- HTTPRoute 生成失败
- Namespace 已冲突
- Terraform 执行失败

平台应：

- 标记资源状态为 `failed`
- 中止后续部署
- 发送通知
- 记录错误详情

## 十五、与部署流程的关系

资源编排不是独立于部署的系统，而是部署前置条件的一部分。

建议流程为：

```text
配置校验
    ↓
资源状态检查
    ↓
如需新增/变更资源则先处理资源
    ↓
资源就绪后进入部署
```

这样可以确保服务部署时运行条件已经满足。

## 十六、一期实现建议

一期建议先支持最核心资源：

- Namespace
- Service
- HTTPRoute
- PVC

并优先打通：

- 资源声明
- 资源变更识别
- 审批
- 资源 apply
- 错误处理

暂不建议一期即实现：

- 复杂 Secret 管理
- 公网/内网多策略路由
- 高级网络策略
- 多集群资源编排

## 十七、总结

NOVA 资源编排模块的核心价值在于：

> 把“上线前需要手工申请和创建资源”的隐性流程，纳入平台治理和自动化执行。

这样可以显著减少运维沟通成本，也让服务接入和发布流程更标准、更可追踪。
