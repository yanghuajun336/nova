# NOVA 权限模型设计

## 一、设计背景

NOVA 平台需要同时服务开发者、运维和后续平台管理员。随着平台承载的能力逐步增加，权限模型会直接影响以下方面：

- 服务接入边界
- 配置修改边界
- 生产发布审批边界
- 服务锁定与解锁边界
- 报表可见范围
- 平台治理能力的安全性

因此，权限模型必须���一开始就设计清楚，即使一期只实现简化版本，也要有明确的演进方向。

## 二、设计目标

权限模型的目标包括：

- 确保不同角色看到和操作的内容不同
- 保护平台治理能力不被随意绕过
- 支持按服务或项目控制权限
- 支持审批和锁定等高风险动作的权限隔离
- 与 WebUI、CLI、API 保持一致
- 支持后续扩展更细粒度权限

## 三、设计原则

### 1. 角色优先，资源细分

一期建议先采用基于角色的访问控制模型（RBAC），后续再逐步增加资源级细粒度控制。

### 2. 高风险操作必须单独授权

例如：

- 审批生产部署
- 解锁服务
- 修改平台默认规则
- 触发资源变更

这些动作不应仅依赖“是否能看见页面”。

### 3. 权限模型与组织结构弱耦合

一期不建议把权限模型深度绑定到复杂组织架构，而应优先支持：

- 角色
- 团队
- 项目
- 服务责任人

## 四、推荐角色

一期建议至少定义以下角色：

### 1. 开发者（developer）

主要职责：

- 创建服务
- 编辑服务的业务配置
- 查看服务、流水线、部署记录
- 发起服务接入和资源申请

### 2. 运维（operator）

主要职责：

- 查看全部服务
- 审批生产发布
- 锁定和解锁服务
- 处理资源变更和失败问题
- 查看平台全局状态

### 3. 平台管理员（admin）

主要职责：

- 管理平台默认值
- 管理模板版本
- 管理 Jira 全局配置
- 管理审批规则
- 管理通知策略
- 拥有全局读写能力

### 4. 只读用户（viewer，可选）

主要职责：

- 查看服务、流水线、报表
- 不允许执行变更操作

## 六、资源粒度建议

除了角色，还建议考虑资源范围。

### 1. 服务范围

开发者默认只能：

- 查看自己负责或所属团队的服务
- 编辑自己负责或所属团队的服务

### 2. 项目范围

运维和管理员可跨项目查看与操作。

### 3. 全局范围

以下能力建议仅管理员可操作：

- 模板版本治理
- 平台默认值管理
- Jira 全局状态映射
- 审批规则模板

## 七、特殊动作权限

### 1. 生产审批

建议仅：

- 运维
- 管理员
- 或明确授权的审批角色

可以执行批准/拒绝操作。

### 2. 服务解锁

服务被锁定通常意味着生产故障风险，因此解锁必须比普通配置编辑更严格。

建议仅：

- 运维
- 管理员

可以解锁服务。

### 3. 资源变更审批

涉及域名、PVC 等资源调整时，建议只允许运维或管理员审批。

## 八、与 WebUI 的关系

WebUI 应根据权限模型控制：

- 菜单可见性
- 按钮可见性
- 表单字段可编辑性
- 页面操作能力
- 审批动作入口
- 锁定/解锁入口

但需要注意：

> 前端权限控制只负责体验，最终仍应以后端 API 权限校验为准。

## 九、与 CLI 的关系

CLI 也应遵循同一套权限模型。

例如：

- 开发者调用 `nova lock unlock` 应被拒绝
- 只读用户调用 `nova approval approve` 应被拒绝
- 管理员才可执行平台全局配置修改命令

错误返回应明确提示权限不足，而不是模糊失败。

## 十、与审计日志的关系

所有高风险操作都必须写审计日志，包括：

- 创建服务
- 修改配置
- 生成 workflow
- 审批通过/拒绝
- 锁定服务
- 解锁服务
- 修改平台规则
- 资源变更审批

这意味着权限模型和审计模型应配套设计。

## 十一、一期建议实现方案

一期建议采用简化 RBAC 模型。

### 推荐做法

- 用户信息从平台登录态获取
- 后端维护用户角色
- 服务记录中保留 owner_team 和 maintainers
- API 校验“角色 + 资源归属”

### 一期无需过早实现

- 复杂组织树
- 多层继承权限
- 细粒度字段级权限
- 动态权限表达式引擎

## 十二、后续扩展方向

后续可逐步扩展：

- 团队管理员角色
- 项目级角色绑定
- 服务级 ACL
- 审批人动态路由
- 与企业组织架构系统对接
- 更细粒度的只读/审批/运维分离角色

## 十三、总结

NOVA 权限模型的核心目标不是“把每个按钮都做权限”，而是确保：

- 平台治理能力不会被轻易绕过
- 高风险动作有清晰责任边界
- 不同角色拥有合适的视图和操作权限
- WebUI、CLI、API 共用同一套权限逻辑

在当前阶段，简化 RBAC + 服务归属控制，是最适合的一期方案。
