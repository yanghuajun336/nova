# NOVA 总体架构设计

## 一、架构目标

NOVA 的总体架构设计需要满足以下目标：

- 支持 Node.js、Python、Go 多语言项目
- 支持 20-50 个微服务统一接入
- 与 Gitea、Jira、Harbor、Kubernetes、SonarQube 集成
- 支持生产审批、自动回滚、服务锁定
- 支持资源自动生成
- 支持邮件通知与周报
- 支持后续 CLI / AI Agent 扩展

## 二、总体架构分层

NOVA 建议划分为四层：

### 1. 交互层

包括：

- WebUI
- CLI
- 后续 AI Agent Skill

负责用户与平台的交互入口。

### 2. 平台服务层

包括：

- 服务管理模块
- 配置校验模块
- 工作流生成模块
- Jira 联动模块
- 审批模块
- 回滚锁定模块
- 资源编排模块
- 报表与通知模块

这是 NOVA 的核心治理层。

### 3. 执行层

包括：

- Gitea Actions Runner
- 中央 Workflow 模板仓库
- 部署脚本
- K8s 资源渲染脚本
- Jira/邮件/回滚执行脚本

这是实际完成 CI/CD 动作的执行层。

### 4. 外部依赖层

包括：

- Gitea
- Jira Cloud
- Harbor
- Kubernetes
- SonarQube
- Terraform / 腾讯云资源

## 四、核心模块设计

### 1. 服务管理模块

负责：

- 注册新服务
- 管理服务与仓库绑定
- 管理运行时语言和版本
- 管理 Jira 项目关系
- 管理资源配置
- 管理当前启用环境

### 2. 配置校验模块

负责：

- 校验 `nova.yml`
- 应用平台默认值
- 拒绝违反平台治理规则的配置
- 输出统一错误信息

### 3. 工作流��成模块

负责：

- 根据 `nova.yml` 自动生成业务仓库入口 workflow
- 根据语言选择对应中央模板
- 管理模板版本
- 管理 workflow 的更新策略

### 4. Jira 联动模块

负责：

- 从分支解析 Issue ID
- 回写 Jira 状态
- 写入 Jira 评论
- 回滚时自动创建 Bug Issue
- 接收 Jira Webhook（如后续需要）

### 5. 审批模块

负责：

- 生产发布审批单创建
- 审批邮件发送
- 审批结果回调
- 审批超时控制
- 审批结果落库

### 6. 回滚与锁定模块

负责：

- 处理生产部署失败后的自动回滚
- 创建 Jira Bug
- 锁定服务，阻止后续继续部署
- 提供运维解锁入口

### 7. 资源编排模块

负责：

- 根据服务声明���成 K8s 资源
- 自动生成 HTTPRoute
- 自动生成 PVC
- 与 Terraform 联动处理云资源

### 8. 报表与通知模块

负责：

- 流水线结果通知
- 部署结果通知
- 回滚通知
- 每周报表邮件
- 统计构建成功率、时长、部署频率等指标

## 五、配置契约：`nova.yml`

`nova.yml` 是 NOVA 平台和业务仓库之间的核心契约，负责描述：

- 服务身份
- 运行时信息
- 构建规则
- 测试规则
- 部署策略
- 资源需求
- Jira 集成关系

平台基于它生成标准化 workflow，并补齐默认规则��

## 六、业务仓库与模板仓库关系

### 业务仓库

业务仓库中建议仅保留：

- 源代码
- Dockerfile
- `nova.yml`
- 平台生成的轻量入口 workflow

### 中央模板仓库

中央模板仓库中维护：

- Node.js 模板
- Python 模板
- Go 模板
- 通用部署脚本
- Jira 脚本
- 邮件脚本
- 回滚脚本
- K8s 模板

这样能实现模板集中治理和统一升级。

## 七、工作流与平台职责边界

### 平台负责治理

- 配置解析
- 默认值策略
- 审批规则
- 服务锁定
- 统计与报表
- 外部系统状态管理

### Workflow 负责执行

- checkout
- 构建
- 测试
- 扫描
- 构建镜像
- 推送镜像
- 部署
- 健康检查
- 回调平台结果

遵循原则：

> 平台做决策，工作流做动作。

## 八、环境与分支映射

建议固定如下映射：

| 分支 / 事件 | 目标环境 | 行为 |
|---|---|---|
| Pull Request | 无 | 仅 CI |
| `feature/*` | 无 | 仅 CI |
| `hotfix/*` | 无 | 仅 CI |
| `develop` | `dev` | 自动部署 |
| `release/*` | `staging` | 自动部署 |
| `main` | `prod` | 审批后部署 |

## 九、命名规范

### Namespace

采用统一格式：

```text
{env}-{project}-{service}
```

例如：

```text
dev-app-iplocator
staging-app-iplocator
prod-app-iplocator
```

### 镜像

建议格式：

```text
harbor.nova.internal/{project}/{service}:{version}-{gitsha}-{date}
```

### 分支

建议格式：

```text
feature/APP-123-add-iplocator
hotfix/APP-456-fix-timeout
```

## 十、技术选型建议

### 后端

建议使用：

- Python + FastAPI

原因：

- 与 API 设计契合
- 校验模型和 OpenAPI 友好
- 适合平台类系统快速迭代

### 前端

建议使用：

- Vue 3 + Element Plus

原因：

- 适合后台管理系统
- 组件成熟
- 和你的倾向一致

### 数据库

建议使用：

- PostgreSQL

原因：

- 结构化能力强
- 支持 JSON 字段
- 适合平台元数据和审计数据

## 十二、结论

NOVA 的架构不是追求功能堆叠，而是优先构建一套：

- 边界清晰
- 配置统一
- 模板集中
- 治理可控
- 易于演进

的平台基础设施。

后续所有具体设计，都应围绕这一总体架构展开。
