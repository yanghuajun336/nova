# NOVA 产品定位

## 一、产品定义

NOVA 是一个面向中小研发团队的 DevOps 平台，围绕多语言微服务场景提供统一的 CI/CD、审批治理、Jira 联动、资源编排与交付追踪能力。

它的目标不是让每个项目独立维护一套流水线，而是通过平台化的方式，让开发者以最少配置完成从代码提交到部署上线的交付流程。

## 二、产品边界

### NOVA 负责什么

NOVA 负责以下能力：

- 服务接入与配置管理
- 平台化 CI/CD 工作流生成
- 统一的测试、质量门禁、构建、部署规则
- Jira 状态回写与联动
- 生产审批流程
- 自动回滚与服务锁定
- 部署资源申请与自动生成
- 邮件通知与统计报表
- CLI / AI Agent 调用入口

### NOVA 不直接替代什么

NOVA 不替代以下系统：

- **Gitea**：代码托管与 Actions 执行平台
- **Jira**：需求、任务、缺陷管理系统
- **Harbor**：镜像仓库
- **Kubernetes**：容器编排平台
- **Terraform**：基础设施资源编排工具
- **SonarQube**：代码质量分析平台

NOVA 是这些系统之上的**统一编排与治理层**。

## 三、目标用户

### 1. 开发者

开发者关注的是：

- 如何快速接入平台
- 如何少写甚至不写 workflow
- 如何通过统一方式完成构建、测试、部署
- 如何让 Jira 与代码、发布自动联动

NOVA 需要为开发者提供尽量简洁的接入体验。

### 2. 运维

运维关注的是：

- 平台规则是否统一
- 资源是否可控
- 生产发布是否有审批
- 回滚是否自动、可追踪
- 服务异常后是否可以快速锁定和处理

NOVA 需要为运维提供可治理、可审计、可配置的管理能力。

### 3. 管理者

管理者关注的是：

- 流水线成功率
- 发布频率
- 回滚次数
- 交付质量趋势
- 问题服务和风险项目

NOVA 需要提供报表、周报和宏观视角的可观测性。

### 4. AI Agent / CLI 使用者

未来 CLI 将服务于 AI Agent，支持：

- 初始化新服务接入
- 查询服务配置
- 生成平台标准化变更
- 与平台 API 协作执行自动化操作

这要求 NOVA 从一开始就具备稳定的配置契约和 API 边界。

## 四、核心价值主张

### 1. 从“项目自己维护流程”转向“平台统一治理流程”

每个仓库不再维护一套复杂 CI/CD，而是由平台自动生成轻量入口文件并引用中央模板。

### 2. 从“靠人记忆规范”转向“靠平台强制规范”

例如：

- 分支命名规范
- Jira 关联规则
- 生产审批规则
- 覆盖率门禁
- 回滚策略
- 服务锁定策略

都不依赖人手工记忆，而由平台落地执行。

### 3. 从“工具分散使用”转向“交付链路打通”

通过 Jira Issue ID 作为交付追踪主键，实现：

- Jira → 分支
- 分支 → 提交
- 提交 → 流水线
- 流水线 → 部署
- 部署 → 回滚
- 回滚 → Jira Bug

形成闭环。

### 4. 从“接入成本高”转向“开发者最小配置”

开发者只需要配置业务相关项，例如：

- 服务名
- 语言
- 构建命令
- 测试命令
- 端口
- 域名
- Jira 项目 Key

而不需要了解全部平台内部实现。

## 五、与现有工具的关系

### Gitea

- 负责代码托管
- 承载 Actions Runner
- 承载中央模板仓库
- 承载业务项目入口 workflow

### Jira

- 负责需求和缺陷跟踪
- 提供项目 Key
- 提供状态流转目标
- 接收平台回写状态和评论
- 回滚时自动创建 Bug

### Kubernetes

- 负责应用运行
- 承载 dev / staging / prod 环境
- 提供 Namespace、Deployment、Service、HTTPRoute、PVC 等资源

### Terraform

- 用于对接腾讯云资源申请与变更
- 为域名、云资源等自动化编排提供基础能力

### SonarQube

- 提供代码质量扫描与门禁
- 作为流水线质量控制的一部分

## 六、产品形态

NOVA 的产品形态建议为：

### 1. WebUI

面向开发者和运维提供：

- 服务接入
- 配置编辑
- 审批处理
- 运行查看
- 锁定解锁
- 报表展示

### 2. CLI

面向自动化和 AI Agent 提供：

- 初始化服务
- 校验配置
- 查询平台对象
- 与 API 协作执行操作

### 3. 中央模板仓库

作为平台执行层，承载：

- 多语言 workflow 模板
- 公共部署脚本
- Jira / 邮件 / 回滚脚本
- K8s 资源模板

## 七、成功标准

如果 NOVA 成功落地，应该达到以下效果：

1. 新服务接入平台的时间显著下降
2. 业务仓库中几乎不再需要手写复杂 workflow
3. 多语言服务通过同一套治理规则执行流水线
4. 生产发布具备统一审批、回滚、锁定机制
5. Jira 与部署链路形成自动同步
6. 研发团队可以通过报表看到交付效率和质量趋势

## 八、未来演进方向

未来可以逐步扩展为：

- 多集群支持
- GitOps 集成
- 更复杂的部署策略
- 平台权限与组织治理
- AI Agent 深度自动化运维
- 智能风险识别与发布建议

但第一阶段必须先把最核心的平台化 CI/CD 能力打稳。
