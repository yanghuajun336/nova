# NOVA WebUI 表单设计

## 一、设计目标

NOVA WebUI 的目标不是把 `nova.yml` 原样展示给用户填写，而是通过分步向导与角色化视图，让不同角色只看到自己真正需要关注的内容。

核心目标包括：

- 开发者最少输入
- 运维可查看完整治理信息
- 表单与 `nova.yml` 一一映射
- 平台自动补全默认值
- 支持可视化生成配置
- 后续支持 CLI / API / YAML 统一模型

## 二、用户角色

### 1. 开发者

开发者主要关注：

- 新服务接入
- 技术栈和构建命令
- 测试命令
- 域名和端口
- Jira 项目绑定

### 2. 运维

运维主要关注：

- 平台治理策略
- 生产审批
- 服务锁定与解锁
- 资源申请和变更
- 风险服务治理

### 3. 平台管理员（后续）

后续可扩展平台管理员角色，负责：

- 模板版本治理
- 默认值策略
- Jira 全局配置
- 审批人配置
- 通知模板配置

## 三、设计原则

### 1. 简化模式优先

默认展示面向开发者的简化表单，不暴露过多平台内部细节。

### 2. 高级模式可切换

运维和高级用户可以切换到高级模式，查看更多配置和平台推导结果。

### 3. 表单生成配置，而不是用户手写 YAML

页面通过表单生成标准化 `nova.yml`，并允许用户预览最终结果。

### 4. 平台强规则只展示不开放修改

例如生产环境审批和自动回滚，应作为平台规则展示，但不应让普通开发者随意修改。

## 四、新建服务流程

建议采用多步骤向导。

---

## Step 1：基础信息

### 字段

- 服务名称
- 项目标识
- Gitea 仓库
- 团队
- 维护人

### 设计建议

- 仓库使用搜索选择器，不允许手填
- 选择仓库后自动检查是否存在 `nova.yml`
- 自动校验当前用户是否对仓库有权限

---

## Step 2：技术栈与构建

### 字段

- 技术栈语言
- 运行时版本
- Dockerfile 路径
- Build Context
- 是否执行构建命令
- 构建命令

### 智能建议

根据仓库内容自动给出推荐值：

- 检测到 `package.json` → 推荐 Node.js
- 检测到 `pyproject.toml` 或 `requirements.txt` → 推荐 Python
- 检测到 `go.mod` → 推荐 Go

---

## Step 3：测试与质量

### 字段

- 是否启用单元测试
- 单元测试命令
- 是否启用接口测试
- 接口测试命令
- 是否启用 E2E 测试
- E2E 测试命令
- 是否启用覆盖率校验
- 覆盖率阈值
- 是否启用 SonarQube

### 默认建议

- 单元测试：启用
- 接口测试：启用
- E2E：启用
- 覆盖率：90
- SonarQube：启用

---

## Step 4：部署策略

### 环境卡片

建议展示为三张环境卡片：

- dev
- staging
- prod

### 展示内容

#### dev

- 启用环境
- 自动部署（默认开启）
- 审批要求（只读）
- 自动回滚（只读）

#### staging

- 启用环境
- 自动部署（默认开启）
- 审批要求（只读）
- 自动回滚（只读）

#### prod

- 启用环境
- 自动部署（只读，默认关闭）
- 审批要求（只读，默���开启）
- 自动回滚（只读，默认开启）

### 设计重点

生产环境的审批和自动回滚属于平台治理规则，开发者只能看到，不应修改。

---

## Step 5：资源配置

### 网络配置

字段：

- 服务端口
- 访问域名

后续可扩展：

- 暴露方式（内网 / 公网）
- TLS 证书策略

### 存储卷配置

支持添加多个卷，每个卷包含：

- 卷名称
- 容量
- 挂载路径

设计为可增减列表。

---

## Step 6：Jira 集成

### 字段

- Jira 项目 Key
- 是否从分支名解析 Issue ID

### 建议

若平台能调用 Jira API，可直接拉取项目列表作为下拉选项，避免手填错误。

---

## Step 7：确认与生成

页面建议同时展示：

- 配置摘要
- ��自动生成的资源列表
- 最终 `nova.yml` YAML 预览

这样用户可以在提交前确认平台将做什么。

## 五、编辑服务页面设计

建议将服务详情页拆分为多个 Tab：

### Tab 1：基础配置

- 服务名
- 项目
- 仓库
- 技术栈
- 构建配置

### Tab 2：测试与质量

- 测试开关与命令
- 覆盖率阈值
- SonarQube 开关

### Tab 3：部署与资源

- 环境信息
- 域名
- PVC
- Namespace 预览
- 最近部署状态

### Tab 4：Jira 与治理

- Jira 项目绑定
- 最近关联 Issue
- 服务锁定状态
- 最近回滚记录
- 生产审批策略（只读展示）

## 六、开发者模式与运维模式

### 开发者模式

显示：

- 基础信息
- 技术栈
- 构建命令
- 测试命令
- 端口 / 域名
- Jira 项目

隐藏：

- 模板版本
- 平台默认值细节
- 服务锁定
- 审批超时
- 内部治理策略

### 运维模式

额外显示：

- Namespace 生成结果
- 审批规则
- 服务锁定状态
- 资源变更审计
- 模���版本信息
- 平台默认值覆盖情况

## 七、表单与 `nova.yml` 的映射

WebUI 应将用户输入映射到标准配置对象，平台补全默认值后生成最终 YAML。

例如：

### 用户输入

- 服务名：iplocator
- 项目：app
- 仓库：app/iplocator
- 语言：nodejs
- 版本：20
- 构建命令：npm run build
- 端口：3000
- 域名：iplocator.internal.example.com
- Jira：APP

### 平台生成

- `metadata`
- `spec.runtime`
- `spec.build`
- `spec.test`
- `spec.deploy`
- `spec.resources`
- `spec.integrations`

## 八、关键交互增强建议

### 1. 仓库自动检测

选择仓库后自动扫描：

- Dockerfile
- package.json
- pyproject.toml
- go.mod
- 测试目录

给出建议配置。

### 2. YAML 实时预览

表单变化时，右侧实时展示生成后的 `nova.yml`。

### 3. 风险提示

在 prod 区域固定展示说明：

- 生产环境必须审批
- 部署失败会自动回滚
- 回滚后自动创建 Jira Bug
- 服务将被锁定，直到修复后手动解锁

## 九、总结

NOVA WebUI 不应设计成“配置中心页面”，而应设计成：

> 一个围绕服务接入与交付治理的产品化操作入口。

它的核心价值在于降低接入门槛、强化统一治理、把复杂度隐藏在平台内部。
