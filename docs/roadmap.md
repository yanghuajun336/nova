# NOVA 实施路线图

## 一、总体思路

NOVA 的建设应遵循“先设计、后实现；先最小闭环、后能力扩展”的原则。

对于当前团队规模和项目复杂度，建议优先构建一个可落地的最小平台闭环，而不是一开始追求大而全。

## 二、阶段划分

建议分为四个阶段推进。

---

## Phase 1：设计基线与基础骨架

### 目标

完成平台设计基线，搭建最基础的结构骨架。

### 工作内容

- 完成产品定位与架构设计文档
- 完成 `nova.yml` v1alpha1 规范
- 完成 WebUI 接入表单设计
- 完成中央 Workflow 模板体系设计
- 创建中央模板仓库
- 创建 NOVA 主仓库
- 规划后端 API 与数据库设计

### 交付物

- 当前设计文档仓库
- 文档版设计基线
- 仓库结构规划

---

## Phase 2：最小可运行平台 MVP

### 目标

打通从服务接入到开发环境部署的最小闭环。

### 工作内容

- 实现 NOVA 后端基础服务
- 实现 `nova.yml` 校验逻辑
- 实现服务接入接口
- 实现 workflow 入口文件自动生成
- 实现 Node.js / Python / Go 三套基础模板
- 打通 Gitea Actions Runner
- 打通 Harbor 镜像构建与推送
- 打通 Kubernetes dev 环境部署
- 实现基础 Jira 评论联动

### 交付标准

- 新服务可通过平台接入
- 平台可自动生成 workflow
- Push 到 develop 可自动部署到 dev

---

## Phase 3：治理能力增强

### 目标

补齐生产治理能力和平台核心控制能力。

### 工作内容

- 引入 SonarQube 质量门禁
- 实现 staging 环境自动部署
- 实现 prod 审批机制
- 实现生产自动回滚
- 实现 Jira Bug 自动创建
- 实现服务锁定与解锁机制
- 实现邮件通知
- 接入资源申请能力（域名 / PVC / HTTPRoute）

### 交付标准

- 生产发布必须审批
- 生产失败自动回滚
- 回滚自动创建 Jira Bug
- 异常服务可被锁定

---

## Phase 4：平台产品化完善

### 目标

完善平台使用体验和统计能力。

### 工作内容

- 完成 WebUI 服务管理页面
- 完成部署与审批界面
- 完成锁定服务管理界面
- 完成流水线历史与部署历史查看
- 实现周报生成与邮件发送
- 实现 CLI 初版
- 为 AI Agent Skill 预留接口

### 交付标准

- 平台具备可视化接入与治理能力
- 可以按服务查看部署、审批、回滚记录
- 可以输出周报和关键统计指标

## 四、实施建议

### 1. 先打最小闭环

建议先选一个 Node.js 服务做试点，完成：

- `nova.yml`
- 自动生成 workflow
- 构建、测试、推送镜像
- dev 环境部署
- Jira 评论回写

### 2. 再扩展到多语言

Node.js 跑通后，再补 Python 和 Go 模板，保持统一抽象。

### 3. 再引入治理能力

试点稳定后，再引入：

- 生产审批
- 自动回滚
- 服务锁定
- 资源申请自动化

### 4. 再做产品化界面

等核心链路稳定后，再逐步建设完整 WebUI 与 CLI，避免前期 UI 先行导致设计频繁返工。

## 五、风险提示

在实施过程中，需重点关注以下风险：

### 1. 模板过度参数化

如果模板参数设计过多，会导致维护成本迅速升高。

### 2. 平台职责过重

如果平台承担过多运行时逻辑，后续排障会非常复杂。

### 3. 外部系统耦合过强

Gitea、Jira、Harbor、K8s、Terraform 的耦合边界需要尽早明确。

### 4. 规则过早定死

一些能力例如数据库迁移、复杂部署策略，应保留扩展点，但不要在一期就做得过重。

## 六、下一阶段文档建议

在当前文档基础上，建议下一步优先补齐：

1. NOVA 后端 API 设计
2. NOVA 数据库表结构设计
3. 审批流设计
4. 资源编排设计
5. CLI / AI Agent Skill 设计

## 七、结论

NOVA 的实施路线应坚持：

- 先统一抽象
- 再打通闭环
- 再加强治理
- 最后产品化完善

只有这样，平台才能在团队当前规模下真正落地，而不是停留在概念或 Demo 层面。
