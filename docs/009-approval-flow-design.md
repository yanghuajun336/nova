# NOVA 审批流设计

## 一、设计背景

根据当前平台设计目标，生产环境部署需要引入人工审批机制。

已确认的前提包括：

- 当前生产发布尚未正式具备审批流程
- 本次平台设计希望补齐该能力
- 生产环境只需要一级审批
- 审批通知通过邮件发送
- 审批通过后继续部署
- 审批拒绝后终止发布
- 审批超时后自动取消本次发布

因此，NOVA 需要建立一套简洁但可治理的生产审批机制。

## 二、设计目标

审批流设计的目标包括：

- 确保生产部署必须经过人工确认
- 审批动作尽量轻量，降低沟通成本
- 审批过程具备可审计性
- 与 Jira、部署记录、邮件通知形成闭环
- 为未来扩展多级审批保留空间

## 三、适用范围

一期审批流仅适用于：

- `prod` 环境部署

以下场景默认不进入审批：

- Pull Request
- feature / hotfix 分支 CI
- `develop` → dev 自动部署
- `release/*` → staging 自动部署

## 四、总体流程

```text
代码合并到 main
        ↓
流水线执行到生产发布前
        ↓
NOVA 创建审批单
        ↓
发送审批邮件
        ↓
审批人点击批准 / 拒绝
   ┌───────────────┬───────────────┐
   │ 批准          │ 拒绝          │
   ↓               ↓               │
继续部署         终止发布          │
   ↓               ↓               │
部署完成         记录审批结果      │
```

若超时未处理：

```text
待审批 → 超时 → 自动取消
```

## 五、审批单内容设计

每个审批单建议至少包含以下信息：

- 服务名
- 项目标识
- 仓库
- 目标环境
- 分支
- Commit SHA
- 镜像版本
- 触发人
- 关联 Jira Issue
- 本次变更摘要
- SonarQube 结果
- 测试通过情况
- 提交审批时间
- 审批超时时间

## 六、审批邮件设计

审批邮件应尽量清晰简洁，让审批人无需登录多个系统就能做判断。

### 邮件主题建议

```text
[NOVA] 生产发布审批：app / iplocator / APP-123
```

### 邮件正文建议包含

- 服务名：iplocator
- 项目：app
- 环境：prod
- 触发人：yanghuajun336
- 关联 Jira：APP-123
- 镜像版本：harbor.nova.internal/app/iplocator:1.0.0-a1b2c3d-20260407
- 测试结果：全部通过
- 代码质量：SonarQube 通过
- 审批截止时间：2026-04-08 14:30:00

并包含两个操作链接：

- 批准
- 拒绝

## 七、审批动作方式

### 方案选择

一期建议采用：

> 邮件链接审批 + 平台落库确认

即审批邮件中附带带签名 token 的链接。

### 链接示例

```text
https://nova.internal/api/v1/approval-actions/{token}?action=approve
https://nova.internal/api/v1/approval-actions/{token}?action=reject
```

### 安全要求

- token 必须带有效期
- token 必须不可伪造
- token 与审批单强绑定
- token 被使用后应失效或不可重复提交

## 八、审批状态机

建议审批单状态如下：

- `pending`：待审批
- `approved`：已批准
- `rejected`：已拒绝
- `expired`：已过期

### 状态流转

```text
pending → approved
pending → rejected
pending → expired
```

审批状态一旦完成，不允许重复流转。

## 九、审批人与权限

当前确认只需要一级审批，因此一期可以简化为：

- 固定审批人
- 或按服务所属团队配置审批人
- 或先由运维统一审批

建议一期先采用：

> 平台统一配置默认审批人列表，由运维或负责人审批。

后续再扩展：

- 按项目配置审批人
- 按服务配置审批人
- 按角色自动路由审批

## 十、审批超时策略

建议默认审批超时时间：

- 24 小时

超时处理逻辑：

1. 将审批状态设为 `expired`
2. 将对应部署标记为取消
3. 记录审计日志
4. 发送超时通知邮件给触发人
5. 在 Jira 中追加评论（可选）

## 十一、拒绝策略

审批被拒绝时，建议：

1. 将审批状态设为 `rejected`
2. 中止本次部署
3. 记录审批意见
4. 通知触发人
5. 在 Jira 中追加评论（可选）

### 审批意见

建议拒绝时允许填写原因，例如：

- 请补充测试结果
- 风险过高，暂缓发布
- 需等待上游服务验证

## 十二、与部署流程的关系

审批不直接执行部署，而是控制部署是否可以继续。

### 流程边界

- Workflow 执行到 prod 部署前
- 调用 NOVA 创建审批单
- NOVA 返回审批状态
- Workflow 等待批准信号或轮询结果
- 批准后继续执行部署
- 拒绝或过期则终止

### 推荐方式

一期可以采用：

- Workflow 定时轮询审批状态
- 平台审批通过后，状态变为 `approved`
- Workflow 检测到后继续

后续可优化为事件驱动。

## 十三、与 Jira 的关系

审批流与 Jira 的联动建议如下：

### 进入审批时

在原 Jira Issue 中写入评论：

```text
NOVA 自动通知：

该服务已进入生产发布审批阶段，待审批通过后执行生产部署。
```

### 审批通过后

可以写入评论：

```text
NOVA 自动通知：

生产发布审批已通过，部署已开始执行。
```

### 审批拒绝后

可以写入评论：

```text
NOVA 自动通知：

本次生产发布审批被拒绝，请根据审批意见调整后重新发起。
```

### 部署成功后

更新状态为：

```text
已完成
```

## 十四、审计要求

审批是生产治理的重要环节，必须保留审计信息。

建议记录：

- 审批单创建时间
- 审批人
- 审批动作
- 审批意见
- 审批来源 IP（可选）
- 审批 token 使用时间
- 对应服务、部署、Jira Issue

## 十五、一期简化原则

一期审批流不建议做得过重。

### 一期需要做的

- 生产环境一级审批
- 邮件通知
- 邮件链接批准/拒绝
- 审批状态落库
- 审批超时处理
- 审批结果通知部署流程

### 一期暂不做的

- 多级审批
- 并行会签
- 条件路由审批
- 审批人动态计算
- 复杂发布日历控制

## 十六、后续扩展方向

后续可扩展为：

- 按服务指定审批人
- 按项目配置审批策略
- 支持多人审批
- 支持审批规则模板
- 支持审批界面操作
- 支持审批统计分析

## 十七、总结

NOVA 审批流的目标不是制造额外流程成本，而是把生产发布风险收敛到一个清晰、轻量、可审计的控制点。

在你当前团队规模下，一级审批 + 邮件通知 + 自动回滚联动，是最合适的一期方案。
